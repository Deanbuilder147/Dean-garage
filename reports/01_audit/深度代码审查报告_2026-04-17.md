## 🔴 严重安全问题

### 1. 硬编码密钥仍然存在
**状态：** .env 文件中已修复，但**仍在 docker-compose 文件中暴露！**

**位置：**
- `/home/agentuser/Dean-garage/docker-compose.yml` 第 11 行:
  ```yaml
  POSTGRES_PASSWORD: mecha_secure_password  # 弱密码且硬编码！
  ```
- `/home/agentuser/Dean-garage/services/combat-service/docker-compose.yml`:
  ```yaml
  POSTGRES_PASSWORD: ***  # 仍然硬编码！
  ```
- `/home/agentuser/Dean-garage/services/online-battle-service/docker-compose.yml`:
  ```yaml
  POSTGRES_PASSWORD: ***  # 仍然硬编码！
  DATABASE_URL=postgresql://deangarage:***@postgres:5432/...  # 密码直接在 URL 中！
  ```

**风险：** 任何能访问代码仓库的人都能知道数据库密码。生产环境部署将被攻破。

**修复方案：**
- 从所有 docker-compose.yml 文件中删除密码
- 仅使用 .env 文件（已加入 gitignore）
- 生成强随机密码

---

### 2. 通过 console.log 在生产环境中泄露信息
**状态：** 生产代码中存在**数十个** console.log 语句！

**发现位置：**
- `/services/online-battle-service/src/app.js` - 记录错误详情
- `/services/online-battle-service/src/middleware/auth.js` - 记录认证失败
- `/services/online-battle-service/src/index.js` - 记录 WebSocket 用户活动
- `/services/online-battle-service/src/routes/rooms.js` - 记录房间操作
- `/services/map-service/src/routes/battlefields.js` - 记录查询错误
- `/services/combat-service/src/database/db.js` - 记录数据库操作

**风险：** 
- 敏感数据（用户 ID、令牌、房间 ID）被记录到标准输出
- 在生产环境中可能暴露用户活动模式
- 堆栈跟踪可能向攻击者泄露内部结构

**修复方案：**
- 使用 Winston 日志库替换 console.log
- 为开发和生产环境配置不同的日志级别
- 永远不要在 production 中记录认证详情或用户活动

---

### 3. 关键端点缺少输入验证
**状态：** 只有 auth-service 有基础验证！

**缺失验证的服务：**
- `/services/hangar-service/` - 创建单位时**无验证**
- `/services/map-service/` - 创建战场时**无验证**
- `/services/combat-service/` - 战斗移动时**无验证**
- `/services/comm-service/` - 房间/聊天消息**无验证**
- `/services/online-battle-service/` - 有 validation.js 但未在所有地方使用

**风险：**
- 通过未验证的输入进行 SQL 注入攻击
- 缓冲区溢出攻击（超大输入）
- 逻辑炸弹（负数、极端值）

**修复方案：**
- 安装 Zod: `npm install zod`
- 为所有路由处理器添加验证模式
- 在数据库查询前进行验证

**发现的问题：**

**修复方案：**
- 在生产环境中移除开发模式回退
- 在所有服务中实现一致的认证中间件
- 完成 combat-service 认证中的 TODO

## 🟠 高优先级问题

### 5. Combat Service 中有硬编码的 JWT 密钥
**位置：** `/services/combat-service/src/routes/battles.js:8`
```javascript
const JWT_SECRET = process.env.JWT_SECRET || 'mecha-battle-auth-secret-key';  // 回退值很弱！
```

**风险：** 如果 JWT_SECRET 环境变量未设置，使用可预测的默认值

**修复方案：** 移除回退值，强制要求环境变量

---

### 6. 数据库文件存储在不安全的位置
**位置：**
- `/services/combat-service/src/database/db.js:9`:
  ```javascript
  const DB_DIR = '/tmp';  # 全世界可写的目录！
  ```
- `/services/map-service/` - 存储在相对于 src 的 `data/` 目录（较好）

**风险：**
- /tmp 在重启时会被清空（数据丢失）
- /tmp 是全世界可读/可写的（安全风险）
- 数据库文件没有设置文件权限

**修复方案：**
- 将数据库文件存储在服务专用的数据目录中
- 设置严格的文件权限（600）
- 使用环境变量配置数据库路径

---

### 7. 错误处理中的静默失败
**位置：**
- `/services/map-service/src/database/db.js:81`:
  ```javascript
  try {
    db.run('INSERT OR IGNORE...', [...]);
  } catch (e) {}  # 静默失败！
  ```
- `/services/combat-service/src/routes/battles.js:54`:
  ```javascript
  } catch (e) {}  # 空的 catch 块！
  ```

**风险：**
- 错误被隐藏，无法调试
- 数据损坏可能被忽视
- 用户无法收到失败反馈

**修复方案：**
- 始终记录错误（使用 Winston）
- 始终向客户端返回错误响应
- 实现正确的错误边界

### 8. 大多数服务缺少健康检查端点
**状态：** 只有 auth-service 和 online-battle-service 有 /health！

**缺失的服务：**
- hangar-service (3002) - 无 /health
- map-service (3003) - 无 /health
- combat-service (3004) - 无 /health
- comm-service (3005) - 无 /health

**影响：**
- Docker 健康检查静默失败
- 无法监控服务健康状态
- Kubernetes 无法进行就绪探针

**修复方案：** 为所有服务添加 /health 端点：
```javascript
app.get('/health', (req, res) => {
  res.json({ status: 'ok', service: 'map-service', timestamp: new Date() });
});
```

## 🟡 中优先级问题

### 9. 端口配置不一致
**问题：**
- 根目录 docker-compose.yml 说：combat=3004, map=3003
- combat-service/docker-compose.yml 说：map=3004, combat=3003（**反了！**）
- online-battle-service/docker-compose.yml 有自己的网络配置

**风险：** Docker 中服务无法通信

**修复方案：** 在单一事实来源中标准化端口配置

---

### 10. 缺少服务发现机制
**现状：** 所有服务使用硬编码的 URL，如：
```javascript
AUTH_SERVICE_URL=http://auth-service:3001
```

**风险：**
- 没有 Docker 网络时本地开发无法工作
- 服务宕机时没有重试逻辑
- 没有负载均衡

**修复方案：** 添加简单的服务发现或一致地使用 Docker 服务名称

---

### 11. 前端有硬编码的 API 路由
**位置：** `/frontend/src/api/client.js:9`
```javascript
baseURL: '/api'  // 仅在使用 Vite 代理时有效
```

**风险：** 没有 Nginx 反向代理配置时，生产构建无法工作

**修复方案：** 使用环境变量：
```javascript
baseURL: import.meta.env.VITE_API_BASE_URL || '/api'
```

---

### 12. 完全没有速率限制
**状态：** 已确认 - **没有任何服务**有速率限制！

**风险：**
- 针对认证的暴力破解攻击
- 对任何端点的 DDoS 攻击
- API 滥用（垃圾房间创建等）

**修复方案：** 为所有服务添加 express-rate-limit：
```javascript
import rateLimit from 'express-rate-limit';
const limiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 分钟
  max: 100 // 每个 IP 每窗口期最多 100 个请求
});
app.use('/api/', limiter);
```
