## 🔧 修复顺序

### 阶段一：P0 关键安全 (立即执行)

#### 1.1 移除硬编码 JWT 密钥 ⭐⭐⭐

**问题描述:**
代码中使用 `process.env.JWT_SECRET || 'fallback-value'` 模式，即使 .env 文件存在，fallback 值仍然留在代码中。

**影响服务:**
- `services/combat-service/src/routes/battles.js`
- `services/comm-service/src/index.js`

**修复步骤:**
1. 移除所有 fallback 硬编码值
2. 添加启动时环境变量验证
3. 服务启动时检查 JWT_SECRET 是否存在

**修复后代码:**
```javascript
// Before (不安全):
const JWT_SECRET = process.env.JWT_SECRET || 'mecha-battle-auth-secret-key';

// After (安全):
const JWT_SECRET = process.env.JWT_SECRET;
if (!JWT_SECRET) {
  console.error('[启动错误] JWT_SECRET 环境变量必须设置！');
  process.exit(1);
}
```

**验证方法:**
```bash
# 1. 检查代码中不再有 fallback 值
grep -rn "JWT_SECRET.*||" services/*/src/

# 2. 测试服务启动 (无 JWT_SECRET 应该失败)
unset JWT_SECRET && node services/combat-service/src/index.js

# 3. 测试服务启动 (有 JWT_SECRET 应该成功)
export JWT_SECRET='test-secret' && node services/combat-service/src/index.js
```

#### 1.2 数据库持久化加固 ⭐⭐⭐

**问题描述:**
Combat-service 使用 sql.js 内存数据库，数据存储在 `/tmp` 目录，服务重启后数据丢失。

**当前状态:**
```javascript
// services/combat-service/src/database/db.js:14
const DB_PATH = process.env.DB_PATH || path.join(__dirname, '../../data');
```

**修复方案:**

**方案 A: 强制使用持久化目录 (快速修复)**
```javascript
// 强制使用持久化目录
const DB_PATH = process.env.DB_PATH || '/var/lib/mecha-battle/combat.db';

// 确保目录存在
const dbDir = path.dirname(DB_PATH);
if (!fs.existsSync(dbDir)) {
  fs.mkdirSync(dbDir, { recursive: true, mode: 0o755 });
}
```

**方案 B: 迁移到 PostgreSQL (推荐，长期方案)**
```javascript
// 使用 PostgreSQL 替代 sql.js
const { Pool } = require('pg');

const pool = new Pool({
  host: process.env.DB_HOST || 'localhost',
  port: process.env.DB_PORT || 5432,
  database: process.env.DB_NAME || 'mecha_battle',
  user: process.env.DB_USER || 'mecha_user',
  password: process.env.DB_PASSWORD,
});

// 验证连接
pool.query('SELECT NOW()', (err, res) => {
  if (err) {
    console.error('[数据库错误] PostgreSQL 连接失败:', err);
    process.exit(1);
  }
  console.log('[数据库] PostgreSQL 连接成功');
});
```

**执行顺序:**
1. 先执行方案 A (快速修复，30 分钟)
2. 后续执行方案 B (需要安装 PostgreSQL 和迁移数据)

**日志分类标准:**

| 类型 | 保留 | 替换为 | 示例 |
|------|------|--------|------|
| 认证失败 | ✅ | winston.error | 用户登录失败 |

| 关键操作 | ✅ | winston.info | 战斗创建、机甲购买 |
| 变量值 | ❌ | 删除 | `console.log('user:', user)` |

**工具脚本:**

#### 2.2 输入验证加固

**问题描述:**
Combat-service 的战斗动作缺少输入验证，可能导致非法操作。

**影响端点:**
- `POST /api/battles/:id/actions` - 战斗动作
- `POST /api/battles` - 创建战斗
- `PUT /api/mechs/:id` - 修改机甲

**修复方案:**
使用 Zod 进行输入验证（auth-service 已实现，可复制）

**示例代码:**
```javascript
import { z } from 'zod';

// 定义战斗动作 Schema
const BattleActionSchema = z.object({
  unitId: z.string().uuid('Invalid unit ID'),
  actionType: z.enum(['move', 'attack', 'skill', 'item']),
  targetId: z.string().uuid().optional(),
  position: z.object({
    x: z.number().int().min(0).max(15),
    y: z.number().int().min(0).max(15),
  }).optional(),
  skillId: z.string().optional(),
  itemId: z.string().optional(),
});

// 在路由中使用
router.post('/battles/:id/actions', async (req, res) => {
  try {
    const validatedData = BattleActionSchema.parse(req.body);
    // 处理验证后的数据
  } catch (error) {
    if (error instanceof z.ZodError) {
      return res.status(400).json({
        error: '验证失败',
        details: error.errors
      });
    }
    throw error;
  }
});
```

#### 3.2 服务间通信认证

**问题描述:**
服务间 API 调用缺少认证，内部网络被攻破后可横向移动。

**修复方案:**
1. 添加服务间 API 密钥
2. 在 middleware/service-auth.js 中验证
3. 所有内部调用添加 `X-Service-Key` 头

**示例代码:**
```javascript
// middleware/service-auth.js
const SERVICE_KEYS = {
  'combat-service': process.env.COMBAT_SERVICE_KEY,
  'hangar-service': process.env.HANGAR_SERVICE_KEY,
  'auth-service': process.env.AUTH_SERVICE_KEY,
};

function serviceAuth(req, res, next) {
  const serviceKey = req.headers['x-service-key'];
  const serviceName = req.headers['x-service-name'];
  
  if (!serviceKey || !serviceName) {
    return res.status(401).json({ error: '缺少服务认证' });
  }
  
  if (serviceKey !== SERVICE_KEYS[serviceName]) {
    return res.status(403).json({ error: '服务认证失败' });
  }
  
  next();
}
```
