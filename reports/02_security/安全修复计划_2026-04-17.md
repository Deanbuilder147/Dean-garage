### 问题 #1: 硬编码密钥

**影响范围：** 3 个 docker-compose 文件  
**修复工作量：** 30 分钟  
**风险等级：** 🔴 严重

#### 修复步骤：

**步骤 1.1: 创建 .env 文件模板**
```bash
# 位置：/home/agentuser/Dean-garage/.env.example
# 内容：
POSTGRES_PASSWORD=
JWT_SECRET=
DATABASE_URL=
AUTH_SERVICE_URL=
MAP_SERVICE_URL=
COMBAT_SERVICE_URL=
HANGAR_SERVICE_URL=
COMM_SERVICE_URL=
```

**步骤 1.2: 生成随机密钥**
```bash
# 生成强密码
openssl rand -base64 32  # PostgreSQL 密码
openssl rand -base64 64  # JWT Secret
```

**步骤 1.3: 修改 docker-compose.yml 文件**

**根目录 docker-compose.yml:**
```yaml
# 修改前:
environment:
  POSTGRES_PASSWORD: mecha_secure_password

# 修改后:
environment:
  POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
```

**combat-service/docker-compose.yml:**
```yaml
# 修改前:
environment:
  - POSTGRES_PASSWORD=***

# 修改后:
environment:
  - POSTGRES_PASSWORD=${POSTGRES_PASSWORD}
```

**online-battle-service/docker-compose.yml:**
```yaml
# 修改前:
environment:
  - DATABASE_URL=postgresql://deangarage:***@postgres:5432/...

# 修改后:
environment:
  - DATABASE_URL=${DATABASE_URL}
```

**步骤 1.4: 更新 .gitignore**
```bash
# 确保以下文件被忽略:
.env
.env.local
*.env
```

#### 验证方法：
```bash
# 1. 检查 .env 文件不在 git 中
git ls-files | grep -E "\.env$"  # 应无输出

# 2. 检查 docker-compose 文件无硬编码密码
grep -r "PASSWORD.*=" services/*/docker-compose.yml | grep -v "\${"  # 应无输出

# 3. 启动服务测试
docker-compose up -d
docker-compose ps  # 所有服务应正常运行
```

**影响范围：** map-service/src/routes/battlefields.js  
**修复工作量：** 15 分钟  
**风险等级：** 🔴 严重

#### 修复步骤：

**步骤 4.1: 修改认证中间件**

**文件：** `services/map-service/src/routes/battlefields.js`

```javascript
// 修改前 (第 14-17 行):
if (!authHeader) {
  req.user = { userId: 1, username: 'test' };
  return next();
}

// 修改后:
if (!authHeader) {
  // 生产环境拒绝未授权访问

  return next();
}
```

### 问题 #5: JWT 密钥弱回退

**影响范围：** combat-service/src/routes/battles.js  
**修复工作量：** 10 分钟  
**风险等级：** 🔴 严重

#### 修复步骤：

**文件：** `services/combat-service/src/routes/battles.js`

```javascript
// 修改前 (第 8 行):
const JWT_SECRET = process.env.JWT_SECRET || 'mecha-battle-auth-secret-key';

// 修改后:
const JWT_SECRET = process.env.JWT_SECRET;
if (!JWT_SECRET) {
  throw new Error('[配置错误] JWT_SECRET 环境变量必须设置！');
}
```

**步骤 5.2: 同时检查其他服务**
```bash
# 搜索所有 JWT_SECRET 使用
grep -rn "JWT_SECRET" services/*/src/
```

**可能需要同步修复的文件：**
- services/auth-service/src/config/index.js
- services/online-battle-service/src/config/index.js
- services/comm-service/src/middleware/auth.js

#### 验证方法：
```bash
# 1. 未设置 JWT_SECRET 时服务应启动失败
unset JWT_SECRET
node src/index.js  # 应抛出错误并退出

# 2. 设置 JWT_SECRET 后服务应正常启动
export JWT_SECRET=$(openssl rand -base64 64)
node src/index.js  # 应正常启动

# 3. 现有 token 应仍然有效（使用相同密钥）
curl -X GET http://localhost:3004/api/battles \
  -H "Authorization: Bearer <existing_token>"  # 应正常验证
```

### 问题 #6: 数据库存储在/tmp 目录

**影响范围：** combat-service/src/database/db.js  
**修复工作量：** 20 分钟  
**风险等级：** 🟠 高

#### 修复步骤：

**文件：** `services/combat-service/src/database/db.js`

```javascript
// 修改前 (第 9-10 行):
const DB_DIR = '/tmp';
const DB_PATH = path.join(DB_DIR, 'combat.db');

// 修改后:
const DB_DIR = process.env.DB_PATH || path.join(__dirname, '../../data');
const DB_PATH = path.join(DB_DIR, 'combat.db');

// 添加目录创建逻辑（在 initializeDatabase 方法中）:
async initializeDatabase() {
  // 确保数据目录存在
  import { mkdirSync, existsSync } from 'fs';
  if (!existsSync(DB_DIR)) {
    mkdirSync(DB_DIR, { recursive: true, mode: 0o700 });
  }
  // ... 原有初始化逻辑
}
```

**步骤 6.2: 更新 docker-compose.yml**
```yaml
# combat-service/docker-compose.yml:
environment:
  - DB_PATH=/app/data/combat.db
volumes:
  - combat_data:/app/data

volumes:
  combat_data:
```

#### 验证方法：
```bash
# 1. 检查数据目录权限
ls -la /app/data  # 应为 700 权限

# 2. 重启后数据应保留
docker-compose restart combat-service
# 数据应仍然存在

# 3. /tmp 中不应再有数据库文件
ls /tmp/combat.db  # 应不存在
```
